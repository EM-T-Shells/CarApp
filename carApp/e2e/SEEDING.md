# E2E Seeding

The Maestro flows assume deterministic data and two known login accounts.
[`scripts/seed-e2e.mjs`](../scripts/seed-e2e.mjs) creates all of it. It is
**idempotent** — re-run it any time; it converges instead of duplicating.

> ⚠️ Uses the **service_role** key, which bypasses RLS. Run it against a
> **test/staging** Supabase project, never production.

## One-time setup

1. **Add the service_role key** to `carApp/.env.local` (gitignored):

   ```
   SUPABASE_SERVICE_ROLE_KEY=eyJ...   # Dashboard → Project Settings → API → service_role
   ```

   `EXPO_PUBLIC_SUPABASE_URL` is already there. Never prefix the service key
   with `EXPO_PUBLIC_` and never commit it.

2. **Seed the static catalog/providers first (if not already).** The E2E
   script seeds the *login-bound* fixtures; the broader Search/Discovery
   data lives in the existing SQL seeds. Run these once in the Supabase SQL
   editor (they're guarded/idempotent):
   - `carApp/supabase/seeds/service_catalog.sql`
   - `carApp/supabase/seeds/providers.sql`
   - `carApp/supabase/seeds/provider_content.sql`

3. **Configure a fixed test OTP** so the auth flow can enter a known code
   instead of a real email/SMS send. In the Supabase dashboard:
   **Authentication → … → set a test OTP** for `test@carapp.dev` and
   `provider@carapp.dev` (e.g. `123456`, matching the code in
   `auth-flow.yaml` / `provider-onboarding.yaml`).
   This is the **only** manual step the script can't do — there is no Admin
   API to set or read a live OTP.

## Run

```bash
npm run seed:e2e          # from carApp/
# or it runs automatically as part of:
./e2e/run-e2e.sh          # set SKIP_SEED=1 to bypass
```

## What gets seeded

| Entity | Detail |
|---|---|
| `test@carapp.dev` | Customer auth user + `public.users` row (same UID) |
| `provider@carapp.dev` | Approved provider (founding, DETAILER), vetting all-approved, `stripe_account_id` set so payout isn't "blocked" |
| Vehicle | Customer's primary — 2022 Honda Civic |
| Service packages | Full Detail ($150), Express Wash ($49) — published |
| Upcoming booking | `confirmed`, scheduled **today** → customer Upcoming **and** provider My Jobs; enables Start Travel |
| Past booking | `completed` (−10d) + 4 before/after photos + paid deposit/balance + paid payout → receipt, Book Again, earnings |
| Message thread | On the upcoming booking + 3 messages (1 unread) |

All fixture rows use fixed `e2e0…` UUIDs (see `ID` map in the script), so the
upserts are stable across runs.

## Cleanup

```bash
npm run seed:e2e:clean      # or: node scripts/seed-e2e.mjs --clean
```

Removes every fixture row (reverse FK order) **and** the two auth users.
Idempotent — safe to run when nothing exists.

## How accounts map to RLS

`public.users.id` **must equal** `auth.users.id` because every policy keys on
`auth.uid() = id`. That's why auth users are created via the Admin API first
(their UID is generated by Supabase), then every fixture references that UID.
Pure SQL can't do this — hence the Node script.
